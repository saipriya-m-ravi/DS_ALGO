Problem	            Type	            Goal
Tushar Party	Unbounded Knapsack	Min cost to reach exact capacity
Buying Candies	Unbounded Knapsack	Max value within budget

Whenever you see:
“exact fill” → minimize cost → DP (min)
“maximize value within constraint” → DP (max)

1] 🎒 0/1 Knapsack
🔹 Problem
You are given: N items
weight[i] ,value[i]
Knapsack capacity = W

👉 Each item:
Either take it (1) or leave it (0). You cannot take it multiple times
Goal: Maximize total value within capacity

🔹 Intuition (Core Idea)
For every item, you have 2 choices:
❌ Don’t take it
✅ Take it (if weight allows)

So at index i, capacity w:

dp[i][w] = max(
    dp[i-1][w],                          # skip item
    value[i] + dp[i-1][w - weight[i]]    # take item
)
🔹 DP Definition
👉 dp[i][w] = max value using first i items with capacity w

🔹 Base Case
dp[0][w] = 0 (no items → no value)

🔹 Code (2D DP)
def knapsack(W, weights, values, n):
    dp = [[0]*(W+1) for _ in range(n+1)]

    for i in range(1, n+1):
        for w in range(W+1):
            if weights[i-1] <= w:
                dp[i][w] = max(
                    dp[i-1][w],
                    values[i-1] + dp[i-1][w - weights[i-1]]
                )
            else:
                dp[i][w] = dp[i-1][w]

    return dp[n][W]

🔹 Example
weights = [1, 3, 4, 5]
values  = [1, 4, 5, 7]
W = 7

👉 Answer = 9
(pick items with weight 3 and 4)

🔹 Optimization (IMPORTANT 🔥)
We only use previous row → can reduce space

1D DP
def knapsack(W, weights, values):
    dp = [0]*(W+1)

    for i in range(len(weights)):
        for w in range(W, weights[i]-1, -1):   # reverse loop
            dp[w] = max(dp[w], values[i] + dp[w - weights[i]])

    return dp[W]

🔴 VERY IMPORTANT (Interview Trap)
Why reverse loop?
for w from W → weight[i]
👉 Ensures each item is used only once
If you go forward → it becomes unbounded knapsack ❌

🔹 Pattern Recognition
Use 0/1 Knapsack when:Each item can be taken once
You see:
“subset”
“pick or not pick”
“maximize/minimize under constraint”

🔥 Must-Know Variations. Once you understand this, these become easy:
Subset Sum
Equal Partition
Count subsets
Minimum subset difference
Target Sum

🔹 Mental Model (VERY IMPORTANT)
At every step, always think:
“If I take this item, what happens?”
“If I skip it, what happens?”
That’s the entire DP.

2] 0/1 Knapsack II (Value-based DP)
🔹 Problem Twist
In normal knapsack DP is based on weight
👉 But here:
Weights can be very large (like 10⁹)
Capacity also large → normal DP won’t work

🔥 Key Idea
👉 Instead of:“Max value for given weight”
We flip it to:“Min weight for given value”

🔹 DP Definition
Let:
dp[v] = minimum weight required to achieve value v

🔹 Goal
Find: Maximum v such that dp[v] ≤ W

🔹 Initialization
dp[0] = 0
dp[others] = INF

🔹 Transition
For each item:

for v from max_value → value[i]:
    dp[v] = min(dp[v], dp[v - value[i]] + weight[i])

👉 Notice:
Backward loop → 0/1 constraint

🔹 Code (Python)
def knapsack_value_based(weights, values, W):
    max_value = sum(values)
    INF = float('inf')

    dp = [INF] * (max_value + 1)
    dp[0] = 0

    for i in range(len(weights)):
        for v in range(max_value, values[i]-1, -1):
            dp[v] = min(dp[v], dp[v - values[i]] + weights[i])

    for v in range(max_value, -1, -1):
        if dp[v] <= W:
            return v

Instead of asking:
“What’s best value for weight ≤ W?”
We ask: “For each value, what’s the cheapest weight needed?”
Then pick best valid value.

🔹 When to Use This? 👉 Use Value-based DP when:
W is large (like 10⁹ ❌)
sum(values) is manageable (like ≤ 10⁵ ✅)

🔥 Pattern Recognition
Scenario	                    Use
Small W	                    Weight-based DP
Large W, small values	    Value-based DP

🔹 Time Complexity
O(N × sum(values))

🔥 Interview Insight
“Since W is large, I’ll reverse DP to value dimension”

🔹 Mental Model
Normal DP → optimize value under weight
This DP → optimize weight for value

🧠 How to Think in Interview (Practical Approach)
Step 1: Start with standard knapsack
You say: “This is 0/1 knapsack. I’ll try DP on weight.”
Then you realize: W is huge ❌
DP [n][W] not possible

👉 This is the trigger moment
Step 2: Say this (VERY IMPORTANT)
“Weight is too large, so I need to reduce state space. Let me see if I can define DP on value instead.”

Step 3: Flip thinking
Instead of: maximize value under weight
Think: for each value → what’s minimum weight needed?

🔑 Why this trick exists (intuition)
Because: You need one dimension small
Either: 
weight small → normal DP
value small → reverse DP
👉 One dimension must be controllable

🎯 Interview Strategy (REALISTIC)
“Standard DP won’t work due to constraints. One known optimization is to reverse DP and track minimum weight for each value.”

🚀 Your Takeaway
Next time you see: Huge capacity (W), Smaller values
👉 Immediately think: “Can I do DP on value instead?”

3] Unbounded knapsack - allowed to use unlimited number of instances of an item.
🔹 Intuition
At each item: You have 2 choices:
❌ Skip it
✅ Take it and stay on same item (since reuse allowed)

🔹 Recurrence
dp[i][w] = max(
    dp[i-1][w],                    # skip item
    value[i] + dp[i][w - weight[i]]   # take item AGAIN
)

👉 Notice:
In 0/1 → dp[i-1][...] Here → dp[i][...] (same row)

🔹 2D DP Code
def unbounded_knapsack(W, weights, values, n):
    dp = [[0]*(W+1) for _ in range(n+1)]

    for i in range(1, n+1):
        for w in range(W+1):
            if weights[i-1] <= w:
                dp[i][w] = max(
                    dp[i-1][w],
                    values[i-1] + dp[i][w - weights[i-1]]
                )
            else:
                dp[i][w] = dp[i-1][w]

    return dp[n][W]

🔹 1D Optimization (Most Important)
def unbounded_knapsack(W, weights, values):
    dp = [0]*(W+1)

    for i in range(len(weights)):
        for w in range(weights[i], W+1):   # FORWARD loop
            dp[w] = max(dp[w], values[i] + dp[w - weights[i]])

    return dp[W]

🔴 CRITICAL DIFFERENCE (Interview Favorite)
Knapsack	    Loop Direction
0/1	            🔙 backward
Unbounded	    🔜 forward

👉 Why?
Backward → prevents reuse
Forward → allows reuse

🔹 Pattern Recognition
Use Unbounded Knapsack when:
“Unlimited supply”
“Can reuse items”
Words like:
coins
rods
cuts
repeats allowed

🔥 Classic Problems (Same Pattern)
These are ALL unbounded knapsack:
Coin Change (min coins)
Coin Change (count ways)
Rod Cutting
Buying Candies (you saw this 👆)
Tushar’s Party (min cost version)

🔹 Mental Shortcut
If you’re confused:
👉 Ask:
“After taking item, can I take it again?”
YES → Unbounded → forward loop
NO → 0/1 → backward loop

🔥 Interview Insight
Most candidates fail here:
They remember formula
But forget loop direction reasoning
If you explain:
“Forward loop because I want to reuse item”
👉 You stand out.

4] Rod Cutting
🔹 Problem
You have: A rod of length N
price[i] → price of rod piece of length i+1
👉 You can cut the rod into pieces
Goal: Maximize total selling price

🔥 Key Insight
At each length:
You can cut into smaller pieces
You can reuse same length again

👉 This is Unbounded Knapsack

🔹 Mapping to Knapsack
Rod Cutting	                Knapsack
Length	                    Capacity
Piece length	            Weight
Price	                    Value
Can reuse cuts	            Unbounded

🔹 Recurrence
For rod length n:
dp[n] = max(
    price[i] + dp[n - (i+1)]
) for all i where (i+1 ≤ n)

👉 Try every possible first cut

🔹 Code (1D DP)
def rodCutting(price, N):
    dp = [0] * (N + 1)

    for length in range(1, N + 1):
        for cut in range(length):
            dp[length] = max(
                dp[length],
                price[cut] + dp[length - cut - 1]
            )

    return dp[N]

🔹 Complexity
Time: O(N²)
Space: O(N)

🔥 Intuition (Very Important)
At each length: “What if I make the first cut of size X?”
Then: Remaining rod = N - X
Solve recursively → DP

🔴 Pattern Recognition
If you see:
“Cut into pieces”
“Maximize profit”
“Reuse allowed”
👉 Think:
Unbounded Knapsack

5] Coin Sum Infinite
Given: Coins array A, Target sum B
👉 You can use unlimited coins
Goal: Count number of ways to make sum B

🔥 Key Insight
Order does NOT matter
[1,2] and [2,1] are the same
👉 This is Unbounded Knapsack (count ways)

🧠 Intuition
At each coin: Either use it Or skip it
But since reuse is allowed: You can use same coin multiple times

🔹 DP Definition
dp[x] = number of ways to make sum x

🔹 Initialization
dp[0] = 1   # one way: take nothing

🔹 Transition
for coin in coins:
    for x in range(coin, B+1):
        dp[x] += dp[x - coin]

🔥 Why this loop order?
Coins outer → avoids counting permutations
Forward loop → allows reuse

🔹 Code (Python)
def coinChangeWays(coins, B):
    dp = [0] * (B + 1)
    dp[0] = 1

    for coin in coins:
        for x in range(coin, B + 1):
            dp[x] += dp[x - coin]

    return dp[B]

🔴 VERY IMPORTANT (Interview Trap)
Loop Order Changes Everything

✅ Correct (combinations)
for coin in coins:
    for x in range(coin, B+1):

❌ Wrong (permutations)
for x in range(B+1):
    for coin in coins:

🔥 Pattern Summary
Problem	                        Goal	                Loop
Coin Change (min coins)	        minimize	            forward
Coin Change (count ways)	    count	                coins outer
Coin Change (permutations)	    count order matters	    sum outer
🧠 Mental Model
Think:
“I am building combinations by gradually adding coins”
Each coin contributes new ways.

🚀 Interview Insight
“Coins in outer loop ensures combinations, not permutations”
